# L089 — Streamlit vs Next.js for OPENGEM Dashboard Frontend: The Final Pick

**Loop**: 089 / 300
**Phase**: 2 — Integration deep dive
**Date**: 2026-06-06
**Verdict**: **ADOPT-V1 (Next.js for public dashboard + Streamlit for internal ops, both shipping in v1)**

---

## What this loop closes

L061 said: Streamlit is the wrong public dashboard, the right ops sandbox. L073 said: Next.js + shadcn + Tremor is the public surface. Both Phase 1 loops pointed to the same verdict. This Phase 2 loop is the *closing argument* — the place where we commit, with concrete reasoning, to building **two** frontends rather than one, and address the "why not consolidate" question.

The forcing question: a one-person-team is going to fight to maintain *one* frontend, let alone two. Are we really sure?

Yes. Below is the case.

## The framing fallacy

The temptation is to pick one tool and contort it. A Streamlit-only OPENGEM ships in 4 weeks but is unembeddable, slow under load, ungoogleable, and never reaches the YouTuber persona. A Next.js-only OPENGEM forces the scenario-authoring workflow into a web form rather than a Python notebook, which is a structural mismatch for the ops team's tasks.

The right framing is *what is each frontend solving for*, not *which tool wins*. Two answers, two tools:

| Surface | Audience | Latency budget | Embedability | Auth | SEO | Tool |
|---|---|---|---|---|---|---|
| **Public dashboard** | Millions of macro-curious anonymous visitors | 100ms first paint | iframe + script SDK | None | Critical | **Next.js** |
| **Internal ops** | 1-3 OPENGEM engineers + the scenario author | 500ms-5s OK | None | OAuth | Irrelevant | **Streamlit** |

Different problems, different tools. Both are right.

## The Next.js case — the public surface

For the public dashboard surface, Next.js wins on six dimensions:

### 1. SEO

OPENGEM's distribution wedge against TradingEconomics is the **country × indicator long-tail SEO** (200 countries × ~30 indicators × 10 horizons = ~60k pages, every one a real URL with real HTML). Static-site-generated by Next.js's SSG/ISR pipeline, each page renders at build time with full meta tags, OG cards, JSON-LD schema markup, and an internal link graph.

Streamlit serves the entire app as a JavaScript-rendered SPA after a Python boot. Google sees a blank page. *Every other technical advantage Streamlit might have is dominated by this single fact.* SEO is not optional for the macro-curious-YouTuber persona who finds OPENGEM via "Indonesia GDP forecast 2026" Google searches.

### 2. Embeddability

The "JSON-block-per-chart" + iframe-embed-in-Substack story (L007, L111) requires components that work outside the OPENGEM domain. Next.js compiles components to standalone JS bundles; we ship them via `<script src="https://opengem.org/embed/...">` and they render anywhere. Streamlit can be iframed but cannot be component-embedded; the iframe is heavy (full Python runtime per embed), brittle (Streamlit Cloud sleep behavior), and unbrandable.

### 3. Bundle size + performance

The public dashboard must render the first interactive panel in under 500ms on a mid-tier laptop. Next.js with code-splitting + ISR + edge caching can hit this routinely. Streamlit's full-page-rerender model can't — every interaction rebuilds the page in Python on the server, blocking on the slowest cached function.

### 4. Component library reach

shadcn/ui + Tremor + Observable Plot + Lightweight Charts + TanStack Table + globe.gl + Kepler.gl is the modern stack covering everything OPENGEM needs (L073, L066, L071, L015, L014, L070). All of it is React-native and Tailwind-stylable. Streamlit's component ecosystem is shallow by comparison — custom Streamlit components require a separate React bundle compiled into an iframe handshake. Wrong abstraction.

### 5. Multi-tenant edge serving

Next.js on Cloudflare Pages / Vercel / Cloudflare Workers serves edge-cached HTML to thousands of concurrent visitors at $0/mo (free tier) to ~$20/mo (Pro). Streamlit's process-per-app model puts a hard ceiling at ~100-300 concurrent users per replica; for the public dashboard we'd need a horizontally-scaled cluster with sticky sessions and a load balancer that costs significantly more for materially worse UX.

### 6. LLM-tool-readability

Every Cursor / Claude / Copilot prompt about React UI in 2026 assumes Next.js + Tailwind + shadcn. AI-assisted dev productivity on this stack is roughly 2x what it is on a non-mainstream framework. For a one-person team, this matters.

## The Streamlit case — the internal ops surface

For the internal ops + scenario authoring workflow, Streamlit wins on three dimensions:

### 1. Scenario authoring is Python-native

L080 already named the scenario engine (copula + parameter sweeps + Monte Carlo) as a Python-heavy workflow. The author writes Python; they want to flip a slider, see the distribution update, save a vintage. That is the *literal canonical* Streamlit use case.

Embedding the same workflow in Next.js means: build a REST endpoint, build a React form, build a chart, wire the form to the endpoint, hot-reload back to the chart. Each step is friction the author doesn't need. Streamlit gives you all of that in 50 lines.

### 2. Single-author, single-session

The ops surface is used by 1-3 engineers on internal authenticated machines. Streamlit's single-process latency model is fine — the author is fine waiting 2 seconds for a chart to re-render after dragging a slider. Public-dashboard latency intolerance does not apply.

### 3. Cost

A Streamlit ops app runs on a single $15/mo Fly.io machine behind a magic-link auth gateway. Total cost of the ops surface: negligible.

## The "consolidate" rebuttal

Three temptations to consolidate, all rebuttable:

### "Use Next.js for ops too — one stack."

The ops scenario authoring would become a series of REST endpoints + React forms + form-state management + Plotly integrations. Each scenario tweak is a 30-minute React-component round trip. Total ops dev cost goes from ~3 weeks (Streamlit) to ~12 weeks (Next.js). For a feature that 3 people use. Net negative.

### "Use Streamlit for the public — one stack."

Already addressed. Public SEO, embeds, and performance are non-starters in Streamlit.

### "Use Observable Framework for everything — one stack."

Observable Framework is excellent for static long-tail country/indicator pages (L066, L091) and absolutely lands in OPENGEM's stack alongside Next.js. But: the *interactive* terminal surface (forecast page with live bands, command palette, vintage drawer drilldown, real-time pulse) is not Observable's strong suit. Build-time data loaders are great for static pages and bad for interactive state.

The right consolidation is **three frontends**: Next.js (interactive public), Observable Framework (long-tail static), Streamlit (internal ops). Each is single-purpose. Each is small. None of them is a battle to maintain individually.

## Cost & timeline

| Surface | Tool | Ramp (v1) | Operating cost |
|---|---|---|---|
| Public dashboard | Next.js + shadcn + Tremor | 6 weeks | $0/mo (Cloudflare Pages) |
| Long-tail explainer | Observable Framework | 2 weeks (after Next.js) | $0/mo (Cloudflare Pages) |
| Ops scenario authoring | Streamlit | 1 week | $15/mo (Fly.io) |
| Ops V&V audit | Streamlit | 1 week | (shared with above) |
| **Total v1 frontend** | | **10 weeks** | **~$15/mo** |

Compare against the consolidated alternatives:
- Next.js-only: ~16 weeks (the ops surface costs 12 weeks instead of 2), same operating cost.
- Streamlit-only: ~8 weeks but ships an unembeddable, unindexable, unscalable public surface that fails the OPENGEM vision in months 2-6 post-launch.

The two-frontend (really, three-frontend) plan is the cheapest *and* the most product-correct.

## Why this is not technical debt

A typical critique: "two frontends doubles the maintenance burden." Look at what each one actually requires:

- Next.js: where 95% of public-user attention lands. Continuous maintenance budget is correct here.
- Observable Framework: builds on a CI schedule, deploys as static artifacts. Maintenance burden: ~1 hour/month.
- Streamlit: internal-only, small surface area, breaks only when the underlying Python library changes (rare). Maintenance burden: ~1 hour/month.

The "single frontend" intuition assumes maintenance scales linearly with framework count. In practice it scales with *surface area of features*. We have three orthogonal feature surfaces; we ship three orthogonal frontends. The combined maintenance cost is *less* than fighting one framework into doing all three jobs.

## Where Streamlit's roadmap matters

`st.fragment` (shipped 2024) is Streamlit's selective-rerun model. If by mid-Y2 it offers true partial reactivity and a real component story, we revisit some specific internal admin pages currently planned for Streamlit. The public dashboard does not move regardless — even with `st.fragment`, Streamlit will not solve SEO, embeds, or sub-100ms latency.

## Risks

1. **Two frontends + one API surface adds API-versioning complexity.** Mitigation: lock the OpenAPI spec (L249) early; both frontends consume it via generated clients.

2. **Skill split.** A Next.js engineer is not always a Streamlit engineer. Mitigation: the one-person team is the Streamlit engineer (Python-native); the Next.js work is the bulk of the public-side feature dev anyway.

3. **Cross-link friction between Next.js and Observable Framework pages.** Mitigation: shared URL convention (L154), shared design system tokens (L146-L148), explicit cross-link components in both frameworks.

4. **The "looks different" problem.** Each frontend has its own CSS. Mitigation: shared design tokens (CSS custom properties exported as a JSON file), strict color/typography rules (L147, L148), regular visual-regression review.
