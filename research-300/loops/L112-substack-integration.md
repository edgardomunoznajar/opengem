# L112 — Substack Integration: API + Paste-In Workflow for the "Share to Substack" Path

**Loop**: 112 / 300
**Phase**: 2 — Deep dive on top candidates
**Date**: 2026-06-06

---

## Thesis of this loop

L007 ranked Substack as the #3 distribution channel for the volume cohort and #4 for the credibility cohort. The reasoning: macro Substack writers (Joey Politano, Skanda Amarnath, Brad DeLong, Adam Tooze, Heather Cox Richardson when she touches econ, hundreds of long-tail mid-tier stacks) are an unusually-leveraged audience — they have readers, they have credibility, *and* they have economic incentive to source good charts because their paid subscribers will churn if posts get sloppy.

The mistake to avoid: treating Substack like Twitter. Twitter's surface rewards bot accounts and constant posting; Substack's surface rewards *one really good thing per week*. The integration strategy can't be "auto-blast our content into Substack." It has to be "make OPENGEM the lowest-friction source of credible charts and numbers for the Substacks already doing the work."

There is no public Substack write-API for posts at the time of writing (mid-2026). Substack's API is read-only — RSS in, but not REST out for post creation. This is an asymmetry we accept and design around.

Verdict: **a three-path workflow: (1) Substack imports `/feeds/daily-digest.rss` as the upstream for `opengem.substack.com`, where OPENGEM publishes a mirror of its weekly digest, leveraging Substack's discovery algorithm; (2) a per-Substack "starter pack" landing page with copy-paste-ready blocks (HTML embed + image URL + cite-this-view text) optimized to drop into Substack's editor in one paste; (3) a `Share to Substack` button on every OPENGEM chart that copies an HTML block to clipboard pre-formatted for Substack's WYSIWYG.** No write-API integration because none exists. The conversion is via UX-of-pasting, not via API automation.

---

## The asymmetry: no Substack write-API

Substack publicly does not offer a "create a post in my publication" REST API. This has been a deliberate choice on Substack's part — they want posts to be authored in their editor, not auto-generated by external tools. Several third-party scraping/headless approaches exist; all are fragile and against Substack's TOS.

OPENGEM accepts this. The integration is **manual on the Substack side, automated on the OPENGEM side** — the writer still hits "publish" in Substack themselves, but every step from OPENGEM's chart → Substack post is as low-friction as we can make it.

What we *do not* attempt:

- Headless Chrome automation against Substack's editor. Fragile, against TOS, breaks every time Substack ships a UI change.
- Scraping Substack's authoring API endpoints. Same reasons.
- Reverse-engineering Substack's mobile API. Same reasons.
- Substack Notes auto-poster. Substack Notes has no public API; even if it did, Notes is the wrong shape for chart-heavy macro posts.

What we *do*:

- Use Substack's RSS-import feature (publicly documented) to mirror our digest into our own Substack publication.
- Use Substack's RSS-import feature to feed OPENGEM updates into our publication's "daily/weekly" newsletter.
- Build paste-optimized HTML blocks that drop cleanly into Substack's editor.
- Build images sized exactly to Substack's content width (700px) so they don't get auto-resized.

---

## Path 1: Substack as a syndication mirror via RSS import

`opengem.substack.com` is a real Substack publication, owned by OPENGEM. Substack lets a publication import an external RSS as the source of posts; we point it at `/feeds/daily-digest.rss` (L106). Substack polls; new items become Substack draft posts; we approve them or let them auto-publish.

The publication's name and aesthetic:

- Name: "OPENGEM Daily" — clear, unambiguous, doesn't pretend to be more than it is.
- Tagline: "The macro-accountability ledger, every morning."
- Cadence: daily, M-F. Weekend digest if anything material happens.
- Pricing: free tier covers everything. There is no paid Substack tier — that violates L006's "never gate the substance" promise. (If we want recurring revenue from email readers, we send to a hosted Buttondown list and charge via OPENGEM proper, not via Substack's Stripe.)
- Cross-recommendations: Substack's "Recommendations" feature is the discovery flywheel — we recommend Joey Politano, Skanda Amarnath, Adam Tooze, Brad DeLong, Calculated Risk, Apricitas, EconBrowser, and ~10-15 more mid-tier macro Substacks; they reciprocally recommend us (we ask). This becomes the cold-start audience.

The RSS-import setup is documented at substack.com/help; it takes ~30 minutes to configure. Items appear in Substack's draft queue within ~15 minutes of feed publication. We start with manual approval (catch any formatting issues); after 30 days of clean delivery, flip to auto-publish.

## Path 2: the per-Substack starter pack

For each of the top ~30 mid-tier macro Substacks (Joey Politano-class), OPENGEM ships a dedicated landing page at `/for/substack/{stack-slug}` (e.g., `/for/substack/joey-politano`). The page contains:

- A personalized greeting acknowledging the writer's recent work (handcrafted, ~50 words).
- Three copy-paste embed blocks: (a) a recession-probability tile for whichever country the writer focuses on, (b) the surprise index for their indicator of choice, (c) a forecast comparison for their indicator vs WEO.
- A "claim your free Studio tier" CTA — $99/mo embed credit, no card required, just sign-in with magic-link.
- A pre-drafted "first post using OPENGEM" template the writer can copy into their next Substack.
- A footer with the embed contract — "Use these in your posts; we ask only that the vintage badge and OPENGEM attribution stay."

These pages are *not* auto-generated. The first 30 are handcrafted by the OPENGEM founder, one per evening over a month. The personalized acknowledgment is the conversion — the writer feels seen, not spammed.

Once the pattern is proven, the next 200 can be templated with personalized variables (name, recent post title, country focus) auto-filled from a CSV.

## Path 3: "Share to Substack" button on every chart

Every interactive chart on OPENGEM has a share button. v1 of the share menu (L155 design loop) lists "Copy link," "Copy as image," "Copy as JSON-block." v2 adds "Copy for Substack."

The "Copy for Substack" action does this:

```html
<!-- copied to clipboard, ready to paste into Substack editor -->
<figure>
  <img src="https://opengem.org/embed/og?kind=forecast&country=USA&indicator=cpi_yoy&theme=editorial&size=banner&w=700"
       alt="US CPI YoY — OPENGEM forecast vs WEO consensus, vintage 2026-06-06"
       width="700" />
  <figcaption>
    US CPI YoY forecast — OPENGEM (3.2%) vs WEO consensus (3.4%), vintage 2026-06-06.
    Source: <a href="https://opengem.org/c/USA/cpi_yoy?v=2026-06-06">opengem.org/c/USA/cpi_yoy</a>
  </figcaption>
</figure>
```

When pasted into Substack's editor:

- The image URL points at OPENGEM's OG-image generator (L111) sized to Substack's 700px content width.
- The image is a 24-hour-cached PNG so it loads in <50ms in the published Substack.
- The figcaption is real text (not part of the image), so Substack's reading-list previews + search index pick up the numbers.
- The hyperlink resolves to OPENGEM with the vintage in the URL — the cite-this-view permanent identifier.

The clipboard payload is `text/html` MIME type, so Substack's WYSIWYG renders it correctly as a figure block instead of plain text.

---

## Image sizing for Substack: pixel-precision

Substack's content area is 700px wide. Embeds wider get squashed; narrower waste real estate. OPENGEM's OG-image generator supports a `w=700` parameter that produces a 700×420 PNG fitting Substack exactly. This single number — 700 — is the difference between a chart that looks intentional in a Substack post and one that looks like it leaked from elsewhere.

Tablet (`w=600`) and mobile (`w=375`) variants are generated automatically and served via `<picture>` with `srcset` when the embed is via an `<img>` tag rather than the iframe loader.

## Reverse-syndication: OPENGEM imports the macro Substack RSS feed

The flip side: OPENGEM's editorial page (`/news`) imports RSS from ~30 macro Substacks (with their permission). This isn't republication — it's a curated "what the macro Substack writers are saying about today's data" feed alongside OPENGEM's own data. The writers get cross-referral traffic; OPENGEM gets editorial gravity it doesn't have to produce in-house.

The implementation is trivial: another Arq job that polls each Substack's RSS at a 4-hour cadence, dedup-checks against a content hash, persists titles + summaries + canonical URLs. The UI renders them as a side-rail on `/news` and on relevant indicator pages.

---

## What we do *not* do via Substack

- **We do not paywall content on Substack.** OPENGEM has no Substack paid tier — ever. Substack subscribers pay $0; if they want to subscribe paid to OPENGEM proper, the path is through opengem.org/pricing, not Substack.
- **We do not write op-ed prose under the OPENGEM name.** The Substack mirror is data-and-charts-first, prose-light. Editorial opinion is not in OPENGEM's brand.
- **We do not auto-post Notes.** Substack Notes is a Twitter clone inside Substack; auto-posting there would be high-volume low-quality and Substack's algorithm would punish it. We post manually if at all.

## Next-step: the `Share to Substack` action snippet

```typescript
// components/share/ShareToSubstack.tsx
export async function copyForSubstack(opts: {
  chartUrl: string;       // canonical chart page URL
  imageUrlBase: string;   // the OG image generator URL prefix
  caption: string;        // human-written caption
  vintage: string;        // ISO date
}) {
  const w = 700;
  const imageUrl = `${opts.imageUrlBase}&w=${w}`;
  const html = `<figure>
  <img src="${imageUrl}" alt="${escape(opts.caption)}" width="${w}" />
  <figcaption>${opts.caption} <a href="${opts.chartUrl}">opengem.org</a></figcaption>
</figure>`;
  await navigator.clipboard.write([
    new ClipboardItem({
      "text/html": new Blob([html], { type: "text/html" }),
      "text/plain": new Blob([opts.caption + " " + opts.chartUrl], { type: "text/plain" }),
    }),
  ]);
  toast.success("Copied — paste into Substack editor");
}
```
