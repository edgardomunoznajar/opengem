# L199 — Trust Signals: Per-Forecast Badges

**Loop**: 199 / 300
**Phase**: 4 — Forecasting product mechanics
**Date**: 2026-06-06

## The decision

A trust badge is a small visual chip attached to a forecast that summarises one binary property (e.g. "vintage-correct" or "PIT-passed"). Badges let a reader assess the quality of a forecast at a glance without reading the methodology page. This loop catalogues every badge, the criterion that earns it, the criterion that revokes it, and the display rules.

The intent is the L001 asymmetry stated visually: trustworthy forecasts wear earned badges; degraded forecasts are marked by *the absence of badges they used to wear*. There is no badge for "trustworthy"; instead there are many narrow, falsifiable badges.

## The badge catalog

| Badge | Criterion to earn | Criterion to lose | Source |
|---|---|---|---|
| **vintage-correct** | The forecast was generated using vintage-correct input data per L182 lineage. | Forecast generated from final-vintage data (back-test reference, not real-time) | L182 |
| **pit-passed** | Cell's PIT-KS p-value ≥ 0.05 over last 12Q rolling | p-value < 0.05 | L193 |
| **pit-passed-12q** | Cell's PIT-KS p-value ≥ 0.05 over last 12Q rolling AND ≥ 0.10 stricter test | p-value < 0.10 | L193 |
| **ensemble-of-N** | Forecast is BMA over N≥3 distinct model variants | Single-model forecast | L189 |
| **beats-AR1** | CRPS strictly less than AR(1) on this cell over OOS window | CRPS ≥ AR(1) | L183 |
| **not-worse-than-WEO** | DM-HLN p-value > 0.10 against WEO on this cell | p-value ≤ 0.10 with OPENGEM losing | L190 |
| **better-than-WEO** | DM-HLN p-value < 0.10 against WEO AND OPENGEM lower CRPS | OPENGEM higher CRPS or p > 0.10 | L190 |
| **mcs-member** | Cell is in Hansen-Lunde-Nason Model Confidence Set at α=0.10 | Not in MCS | L184 |
| **peer-replicated** | Backtest results reproduced byte-identically by ≥1 external third party via L185 harness | No external replication | L185 |
| **replay-verified-7d** | Replay-and-diff CI succeeded on this envelope within 7 days | CI failure | L186 |
| **high-coverage** | Forecast available for ≥80% of Tier-V Core countries at this `(indicator, horizon)` | Coverage drops < 80% | L194 |
| **academic-citation** | Forecast or methodology cited in ≥1 peer-reviewed paper | n/a (sticky once earned) | L008 |
| **early-anticipator** | Forecast moved toward truth before scoring period (L192 trajectory check) | Drift-toward-error pattern | L192 |
| **calibration-warning** | PIT-KS failure (the *absence* of pit-passed is shown explicitly) | n/a (warning, not earned) | L193 |

| **third-party-submission** | Forecast was generated by an external model via L185 harness, vetted | revocation if envelope fails | L185 |

## Two badge classes

The catalog is split into:

- **Positive badges** (earned) — green chip, shown when criterion holds.
- **Warning badges** (revealed) — amber chip, shown when a failure mode is present.

The forecast carries a `trust_badges` array (L181 schema), populated by the backtest engine + lineage check + coverage check on each publication.

```json
"trust_badges": [
  {"id": "vintage-correct", "class": "positive", "earned_at": "2026-06-06T08:00:00Z"},
  {"id": "ensemble-of-3", "class": "positive", "earned_at": "2026-06-06T08:00:00Z"},
  {"id": "pit-passed", "class": "positive", "earned_at": "2026-06-06T08:00:00Z"},
  {"id": "not-worse-than-WEO", "class": "positive", "earned_at": "2026-06-06T08:00:00Z"},
  {"id": "beats-AR1", "class": "positive", "earned_at": "2026-06-06T08:00:00Z"},
  {"id": "mcs-member", "class": "positive", "earned_at": "2026-06-06T08:00:00Z"}
]
```

## Display rules

The dashboard chart's trust panel (L195) shows up to **4 badges** by default, prioritised:

1. `peer-replicated` (rarest, most credibility-conferring) — always shown if present.
2. `replay-verified-7d` — always shown if present (operational trust).
3. `pit-passed-12q` or `pit-passed` — calibration is load-bearing.
4. The highest-tier badge among `beats-AR1`, `not-worse-than-WEO`, `better-than-WEO`, `mcs-member`.

Hover any badge → tooltip with criterion + verification link. Click → opens badge page with full criterion definition + a list of all currently-earning forecasts.

**Warning badges always show**, regardless of slot count. A forecast with `calibration-warning` shows that prominently — we never hide failure to make the trust strip look better.

## The badge page

Each badge has its own page (`opengem.org/badges/{badge_id}`) with:

## Badge provenance

Each badge's "earned at" timestamp is recorded with the underlying evidence:

```json
{
  "badge_id": "pit-passed",
  "earned_at": "2026-06-06T08:00:00Z",
  "evidence": {
    "test": "ks_one_sample_uniform",
    "test_statistic": 0.062,
    "p_value": 0.42,
    "threshold_alpha": 0.05,
    "n_observations": 168,
    "evaluation_window": "2014-Q1..2025-Q4",
    "computed_by_run": "vv_run_2026-W23"
  }
}
```

Every badge is verifiable from primary evidence. A reader can audit any badge claim by running `opengem replay --vv-cell C-03 --country USA --verify-badge pit-passed`.

## Anti-vanity rules

Some badges are tempting to inflate but disciplined against:

- **No "popular" badge.** Pageview counts do not earn badges.
- **No "endorsed by" badge.** Personal endorsements do not earn badges.
- **No badge from social-media counts.** GitHub stars are not a badge.

Every badge is *technical, falsifiable, replayable*.

## Third-party badge submission

A reader can earn badges *for OPENGEM* by independently verifying. The `peer-replicated` badge requires a third party to:

1. Pull a vintage envelope (L186).
2. Replay it byte-identically.
3. Submit a verification claim to `POST /v1/verifications` with the third party's identity + the matching envelope hash.

OPENGEM verifies the submission and attaches the badge with the third party's name (e.g. "peer-replicated by Acme University Macro Group, 2026-04-15").

## The badge inventory page

A public page `opengem.org/badges` lists all badges with currently-earning counts, recent badge changes (badge added / lost events), and the "all-badge-earning forecasts" leaderboard.

## Edge cases

- **Forecast for a Tier-T country with no leaderboard** — no `beats-AR1` / `better-than-WEO` badges (no comparable benchmarks). Shows only badges that don't require comparison.
- **First-vintage of a fresh model** — most badges unearned until 8+ quarters of OOS. Shows `vintage-correct`, `ensemble-of-N`, but not yet `beats-AR1`.
- **Calibration warning that resolves** — warning badge removed; positive badges re-earned.

## API

```
GET /v1/badges                       → catalog
GET /v1/badges/{badge_id}            → badge definition + currently-earning list
GET /v1/forecasts/{forecast_id}/badges → badges on that forecast
POST /v1/verifications               → third-party replication submission
```
